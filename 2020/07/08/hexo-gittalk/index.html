Hexo评论系统配置: Gitalk

发表于2020-07-08|更新于2020-10-15|Hexo 配置手册
|字数总计:915|阅读时长:3分钟|阅读量:

本文将介绍使用 Hexo 配置 Gitalk评论系统，我使用的主题是 hexo-butterfly ，目前已经支持 Gitalk。

如果你使用的主题还不支持 Gitalk，可以考虑自己改一个出来，Gitalk 的官方仓库: gitalk/gitalk，里面有一些基本教程。

我的网页部署在 Github Pages 上面，如果你使用 Gitee 或者 ECS ，理论上是一样的，只是一些参数设置可能稍有不同。

Gitalk 简介

Gitalk 是一个评论系统，它的原理是借助 GitHub Issue 的评论功能，来存储和管理评论。

所以我们首先需要一个 Github仓库 来提供 Issue 的存储，你可以选择新建一个，如果你使用 Github Pages 来部署网页，那么可以像我一样直接把两个仓库合在一起用。

举个例子

Github Pages 地址: github.com/nonlinearthink/nonlinearthink.github.io

Issue 仓库地址: github.com/nonlinearthink/nonlinearthink.github.io/issues

GitHub Pages + Gitalk ，完全使用 Github 的免费服务就可以拥有一个漂亮的个人博客。

环境说明

  • 博客框架: hexo
  • 主题: hexo-butterfly
  • 博客服务: Github Pages
  • 博客地址: nonlinearthink.github.io

创建 OAuth Application 的过程所有 hexo 主题都是通用的，后面的配置文件主题之间有所差异。

选择或者创建 Github 仓库

创建仓库就不教了，注意开启 Issue 功能。

创建 OAuth Application

首先，先跳转到 OAuth Apps 的创建界面。

具体路径是 Settings-Developer settings-OAuth Apps。

点击 Register a new application，创建一个新的 OAuth Application。

以下是对这些字段的说明

表单字段 说明
Application name 必填 应用名字，随便填
Homepage URL 必填 填写你的博客首页地址
Application description 选填 应用描述，随便填
Authorization callback URL 必填 如果有其他的域名的话，就填那个域名，没有的话，和 Homepage URL 一样就行了

填写完之后，会显示这样一个界面，其中 Client ID 和 Client Secret 很重要，待会要用，这里我防止自己的应用被滥用，就先打码了。

修改 butterfly 配置文件

之前的步骤所有的 Gitalk 配置都是一样的，接下来的配置只针对 butterfly 主题，主题与主题之间可能有些不一样。

打开 _data/butterfly.yml，找到 gitalk 的配置项。

把之前获取 Client ID 和 Client Secret 分别填到 client_id 和 client_secret 里面。

接下来要填写你存储 Issue 的仓库，我存储 Issue 的仓库和博客仓库是同一个，如果你使用 Gitee 或者 ECS，就填写你存储 Issue 的仓库就好了，不要因为我这里和博客仓库是同一个就弄混了。

repo 就填写仓库名，nonlinearthink.github.io。

owner 和 admin 不要弄混，owner 是你前面 创建 OAuth Application 使用的账号，admin 是仓库的拥有者或者协作者。

关于其他的字段解释可见官方博客

更新、发布

存留的问题

有时候会出现 Error: Network Error 的报错，猜测可能是 Github 网络的问题，如果大家想要评论却发现不能评论，可以去我 Github 的 Issue 仓库 中直接评论。

针对 hexo-butterfly 3.0 的更新内容

更新内容见官方文档 https://demo.jerryc.me/posts/ceeb73f/#評論。

更新了 3.0 之后发现 gitalk 失效了，需要一些额外的配置。
